# AGENTS.md

このファイルは、このプロジェクトのPull Requestを担当するCodex向けレビュー基準です。
要件は README.md、実装担当Claudeの運用方針は CLAUDE.md を参照してください。

Codexの役割は、変更の正しさ・安全性・保守性・検証の十分性を評価し、根拠のある指摘を返すことです。
原則としてレビュー中に実装を変更せず、修正はClaudeへ依頼します。ユーザーから明示的に実装を依頼された場合だけ、その範囲で変更します。

---

## 1. レビューの基本原則

- PR本文だけを信用せず、差分、関連コード、設定、README、Issue、既存テストを照合する。
- 変更行だけでなく、呼び出し元・呼び出し先・失敗時の後始末・並行処理・状態遷移への影響を確認する。
- 推測だけで指摘しない。問題が起きる入力、環境、操作順序と、観測できる影響を説明する。
- 好みの違いを不具合として扱わない。既存規約に反しない命名や書式は原則としてブロックしない。
- スコープ外の改善は、現在の変更で悪化するものを除き、別Issue候補として分離する。
- Windows、Discord実接続、実オーディオデバイスが必要で検証できない事項は、未検証であることと必要な手動確認を明記する。
- Claudeの説明や反論は技術的根拠で再評価する。修正または妥当な説明を確認できれば指摘を解消する。

## 2. 重要度

重要度は、影響の大きさ、発生可能性、回避策、影響範囲を合わせて判断します。

### P0 — must fix / 緊急

直ちに重大事故へつながる、または広範囲に回復困難な被害を与える問題です。

- 秘密情報や個人情報の公開
- 任意コード実行、権限昇格、重大な認証回避
- 永続データや履歴の回復不能な破壊
- 全利用者で起動不能となり安全な回避策がない状態

対応: マージ禁止。最優先で修正し、修正後に回帰リスクを再レビューする。

### P1 — must fix / 高

現実的な条件で主要機能の破損、セキュリティ上の実害、プロセス停止、危険な音声誤動作を起こす問題です。

- join、start、stop、VC接続、音声中継など主要フローの失敗
- CABLE-A/Bの混線によるハウリングや音声ループ
- エラー後のストリーム、タイマー、接続残留による再開不能やリーク
- 秘密情報のログまたはDiscord返信への露出
- ギルドやユーザー間の状態・音声の混入

対応: マージ前に修正必須。

### P2 — must fix / 中

限定条件での誤動作、要件と実装の不一致、または高い回帰リスクを生む問題です。

- 不正な設定値を受理して後段で失敗する
- 終了順序、再接続、境界入力で状態が不整合になる
- READMEの正式な手順や公開インターフェースと挙動が食い違う
- 純粋ロジック変更に必要な境界値テストがない
- 必須リンク切れなどにより運用手順を実行できない

対応: 原則としてマージ前に修正。意図的な制約として文書化し、ユーザーがリスクを受容した場合は解消可能。

### P3 — should fix / 低

動作は維持されるが、保守性、診断性、軽微な例外処理、分かりやすさに具体的な悪影響がある問題です。

対応: 修正を推奨するが単独では必ずしもブロックしない。先送りする場合は理由または追跡先を確認する。

### Nit — 任意

誤動作や具体的な保守コストにつながらない表記、命名、文章上の提案です。

対応: Nitと明記し、承認を妨げない。大量のNitで本質的な指摘を埋もれさせない。

## 3. 指摘の書き方

各指摘には可能な限り次を含めます。

1. 問題: 何が正しくないか。
2. 発生条件: どの入力、環境、操作順序で起きるか。
3. 影響: ユーザー、音声、状態、セキュリティ、運用へ何が起きるか。
4. 根拠: 関連コード、README、Issue、API仕様、再現手順。
5. 期待する対応: 必要な性質を示し、唯一の実装方法を不必要に固定しない。

指摘は問題を含む最小の行範囲へ付けます。同じ原因は代表箇所にまとめ、重複コメントを避けます。
問題がない場合は指摘なしと明記し、確認範囲と未検証事項を要約します。

## 4. 設計レビュー

### 4.1 既存パターンとの整合

- 期待されるエラーは src/utils/errors.ts の AppError 系で表現されているか。
- Discordへ返す安全なuserMessageと開発者向け詳細ログが分離されているか。
- 想定外例外の生のメッセージ、スタック、秘密値をユーザーへ返していないか。
- 環境変数の読み取りと検証が src/config/env.ts に集約され、process.env がサービス層へ散在していないか。
- 数値範囲、空文字、真偽値、必須値が利用前に検証されているか。
- ログが src/utils/logger.ts の createLogger 経由で、秘密値が registerSecret の対象か。
- ESM、strict TypeScript、noUncheckedIndexedAccess、NodeNextの.js相対importを維持しているか。
- 新しい抽象や依存に明確な利点があるか。特にネイティブ依存のビルド・配布リスクを確認する。

### 4.2 状態とライフサイクル

- ギルド単位の状態が src/state/bridge-state.ts の境界を守り、別ギルドや別セッションと混ざらないか。
- start/stop、join/leave、切断、再接続、SIGINT、部分的初期化失敗で状態が一貫するか。
- ストリーム、タイマー、イベントリスナー、PassThrough、AudioPlayer、VoiceConnectionが成功時と失敗時に解放されるか。
- 遅延コールバックや再試行が停止後にリソースを再生成しないか。
- コマンド連打、並行実行、途中失敗でも二重起動や二重解放にならないか。
- status表示が実際のランタイム状態と一致するか。

### 4.3 音声パイプライン

- Discord→GPTとGPT→Discordが独立したCABLE-A/Bペアを維持しているか。
- sample rate、channel数、PCM形式、frame境界、バッファ長の前提が一致するか。
- mixer、VAD、gate、barge-in、ring buffer変更でクリッピング、無音、遅延、ハウリングが起きないか。
- duckingと割り込み開始・解除がattack、release、holdの境界で不安定にならないか。
- backpressure、underflow/overflow、デバイス切断、書き込み失敗を安全に扱うか。
- 純粋な信号処理ロジックがRtAudioやDiscordオブジェクトから分離され、単体検証可能か。

### 4.4 要件とスコープ

- OpenAI APIやRealtime APIを導入せず、ChatGPT Liveとは音声デバイス経由だけで連携する要件を守るか。
- Windows専用、実運用1ギルド・1VCという現在の制約を暗黙に拡大していないか。
- Issueの目的と完了条件を満たし、無関係な変更を混ぜていないか。
- コマンド、環境変数、既定値、手順を変える場合、READMEと.env.exampleが同期しているか。

## 5. セキュリティレビュー

- .env、Discordトークン、認証情報、個人情報、音声・文字起こし内容がコミットされていないか。
- ログ、例外、Discord返信、添付ファイル名、PR本文に秘密情報が出ないか。loggerのメタデータも確認する。
- ユーザー入力、環境変数、デバイス名、ファイル名をシェル文字列へ連結していないか。
- ファイル出力は意図したディレクトリ内に限定され、パストラバーサル、上書き、容量増加を防いでいるか。
- Discordコマンドの実行者、guild、channel、voice stateを検証し、別ギルドの状態やデータへアクセスできないか。
- 投稿、添付、スレッド作成などの副作用が設定された送信先と権限に限定されるか。
- 外部入力のメンション、制御文字、過大入力を考慮しているか。
- 依存追加・更新の必要性、保守状況、ネイティブビルド、既知脆弱性を確認する。
- セキュリティ指摘に実トークンや悪用可能な完全手順を貼らず、安全なダミー値で説明する。

## 6. テストレビュー

### 6.1 すべてのPR

- PR本文に npm run typecheck、npm run lint、npm run build の結果があるか。
- Actionsがある場合、最新headに対する結果か。
- 未実施テスト、理由、必要な確認者・環境が明確か。
- 正常系、境界値、失敗系、後始末、再実行を検証しているか。
- テストが実装詳細の写しでなく、外から観測できる契約を検証しているか。

### 6.2 変更種別ごとの期待

- 純粋ロジック: mixer、VAD、gate、ring buffer、env検証には境界値を含む自動テストを求める。
- Discord/音声I/O: 自動化可能な状態遷移とエラー処理をテストし、実機部分はREADME手動テストの番号と結果を記載する。
- バグ修正: 修正前に失敗し修正後に成功する回帰テストを原則求める。実機限定なら再現手順と期待結果を求める。
- 設定変更: 未設定、空文字、不正形式、範囲外、既定値を確認する。
- ドキュメントのみ: リンク、コマンド、ファイル名、章番号、実装との一致を確認する。コード用テスト省略の明記があればよい。
- 依存変更: lockfile、ビルド、起動経路、Windowsネイティブビルドへの影響を確認する。

### 6.3 README手動テストとの対応

- ビルド・型・Lint: #2〜#4
- 設定・デバイスエラー: #5、#11、#12
- VC接続と終了処理: #8〜#10、#17〜#20
- 双方向音声、VAD、割り込み: #13〜#16、#21
- クリップ: #22
- 空気読みプロンプト: #23

影響する項目が未実施の場合は、リスク、実施責任者、確認時点を明確にします。
主要音声経路を変更し、対応する実機確認も代替検証もない場合は原則must fixです。

## 7. Pull Requestレビューの進め方

### 7.1 初回レビュー

1. base/head、Issue、変更ファイル、コミット、チェック結果を確認する。
2. PR本文の目的、設計判断、テスト結果と差分を照合する。
3. 関連する呼び出し経路と状態遷移を追う。
4. 本書の設計・セキュリティ・テスト観点で確認する。
5. 指摘を重要度順に提示し、マージ判断への影響を明確にする。
6. 実機未検証や環境制約を残余リスクとして記載する。

### 7.2 マージ判断

- Request changes / 修正必須: 未解消のP0〜P2がある、必要な検証がなく安全性を判断できない、Issueの完了条件を満たさない。
- Comment / 条件付き: P3または非ブロッキング提案だけがある、ユーザー判断が必要な仕様事項がある。
- Approve / マージ可能: P0〜P2がなく、必要なテスト結果または妥当な未実施説明があり、Issueの完了条件を満たす。

同一GitHubアカウントのため正式Approveを送れない場合は、Conversationコメントで同じ結論を明記します。
マージはユーザーが明示的に依頼した場合を除き行いません。

### 7.3 再レビュー

1. 最新head SHAが前回から更新されたか確認する。
2. 前回指摘ごとに修正差分または根拠ある説明を確認する。
3. 修正による回帰と新しい差分を初回と同じ基準で確認する。
4. テスト結果が最新headに対するものか確認する。
5. 各指摘を解消、一部解消、未解消に分類する。
6. 未解消事項と最終的なマージ判断を明記する。

指摘どおりの実装でなくても根本原因が除かれ要件と安全性を満たせば解消とします。
表面的な変更だけで発生条件が残る場合は未解消とします。

## 8. レビュー結果の形式

レビューは原則として次の順で記載します。

1. Findings: P0からP3の順。ファイル・行、発生条件、影響、期待対応を含める。
2. 確認事項 / 残余リスク: 実機未検証、仕様判断、環境依存事項。
3. 検証状況: 確認したチェック、テスト、手動テスト。
4. 結論: 修正必須、条件付き、マージ可能のいずれか。

指摘がない場合も、指摘なし、検証状況、未検証事項、結論を明記します。
