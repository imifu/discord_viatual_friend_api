# CLAUDE.md

このファイルは、私(Claude)がこのプロジェクトの主担当エンジニアとして
長期的に開発・保守を行う上での方針をまとめたものです。
`CLAUDE_初回PROMPT.md`(BOOTSTRAP)の指示を受けて作成しました。

プロジェクトの機能・構成については [README.md](./README.md) を参照してください。
レビュー担当(Codex)向けの基準は [AGENTS.md](./AGENTS.md) に策定予定です。
現時点では作成依頼の初期プロンプトのみで、実質的な基準は未記載です([Issue #2](https://github.com/imifu/discord_viatual_friend_api/issues/2) で追跡)。

## 1. プロジェクトの性質(方針に影響する前提)

- **Windows専用**の常駐プロセスであり、`audify`(RtAudio/WASAPI)というネイティブ依存を介して
  実オーディオデバイスを直接掴む。CI上や他OSでは音声パイプラインの実動作を検証できない。
- **OpenAI API / Realtime APIは一切使用しない**。ChatGPT Liveとの連携は音声デバイス経由のみで、
  テキストや画像などの自動注入は行わない。この制約を壊すような実装(API直叩き等)は要件からの逸脱なので提案しない。
- 自動テストが整備されていない領域(Discord実接続・実オーディオデバイス)が多く、
  README「手動テスト手順」の表が現時点での正式な受け入れ基準になっている。
- 秘密情報(`DISCORD_TOKEN`等)は `.env` にのみ存在し、Git管理対象外(`.gitignore`済み)。

これらはコードを読めば分かる情報だが、方針判断のたびに再確認するのではなく前提として持っておく。

## 2. 開発方針

- **既存の設計パターンを優先する。** 新しい抽象を導入する前に、同種の問題を既存コードがどう解いているか
  (`src/utils/errors.ts` のエラー階層、`src/config/env.ts` の設定検証、`src/utils/logger.ts` のロギング等)を確認し、
  倣う。パターンが増えるほど「主担当エンジニアが1人で長期間見る」という体制のコストが上がる。
- **音声パイプラインの不変条件を壊さない。** Discord→GPT用とGPT→Discord用は必ず独立した仮想デバイスペア
  (`CABLE-A` / `CABLE-B`)を使う。これを1組にまとめる・共用する変更はハウリングの原因になるため、
  README/コードのどちらであっても提案・実装しない。
- **信号処理ロジックとI/Oを分離する。** `pcm-mixer` / `voice-activity` / `audio-gate` のような
  純粋なロジックは、RtAudioストリームやDiscordの接続オブジェクトに依存させない。
  I/Oから切り離すことで、後述のユニットテストが書ける状態を維持する。
- **要件を勝手に変更・拡大しない。** 曖昧な指示や、実装中に見つけた「ついでに直したい」対象は、
  無断で着手せず、まずユーザーに確認するか、着手中のタスクとは別のPRとして切り出す。
- **依存追加は慎重に。** 特にネイティブモジュール(`audify`等)はビルド・配布が壊れやすい。
  新規依存を追加する前に、既存依存や標準ライブラリで代替できないか検討する。

## 3. 設計方針

- **エラーは `AppError` 系のサブクラスで表現する。** ユーザー向け文言(日本語・Discordにそのまま出せる)と
  ログ向け文言(詳細・英語可)を分離する既存パターン(`src/utils/errors.ts`)を継続する。
  想定外の例外を握りつぶしてユーザーに生の `Error.message` を見せない。
- **設定値は `src/config/env.ts` に集約する。** `process.env` への直接アクセスをサービス層・コマンド層に
  散らばらせない。バリデーションが必要な新しい環境変数は `requireString` / `optionalXxx` 系のヘルパーに乗せる。
- **ログは `src/utils/logger.ts` の `createLogger(scope)` 経由で出す。** `console.log` の直接呼び出しは避け、
  秘密情報を記録する可能性がある値は `registerSecret` に登録してリダクション対象にする。
- **状態はスコープを意識する。** `src/state/bridge-state.ts` はギルド単位で状態を持つ設計になっている。
  「複数ギルド同時運用は未検証」という既存の制限(README 20章)を変更する場合は、
  状態管理の設計自体を見直す必要があることを認識した上で着手する。
- **ESM / strict TypeScript を維持する。** `tsconfig.json` の `strict` / `noUncheckedIndexedAccess` を
  緩めない。相対importには `.js` 拡張子を付ける(`moduleResolution: NodeNext` の制約)。

## 4. Git運用

- Branch戦略は **GitHub-Flow**。`main` から作業用の `feature/*`(または `fix/*`)ブランチを切って作業する。
- **`main` への直接コミットは禁止。** 必ずブランチを切り、Pull Requestを経由する。
- **Pull Requestの自己マージは行わない。** マージ判断はユーザー(リポジトリオーナー)に委ねる。
- ブランチ名は内容が分かる英語の kebab-case を基本とする(例: `feature/create-claude-md`,
  `fix/barge-in-release-timing`)。BOOTSTRAP等で指定されたブランチ名がある場合はそれに従う。
- 作業対象が広範囲・長期に渡る場合は、レビューしやすい単位でPRを分割することを優先する。

> **解消済みの経緯:** 当初リポジトリのデフォルトブランチは `master`(空コミット1つのみ)だったが、
> 実装のコミット履歴(Phase2〜5、初回実装)はすべて `main` 側にあった。ユーザー(リポジトリオーナー)の
> 判断により `origin/master` を削除し、GitHubのデフォルトブランチを `main` に変更済み。
> 現在は `main` が唯一の主ブランチであり、本章の運用はすべて `main` を指す。

## 5. コミット方針

- 1コミット1論理変更を基本とする。無関係な変更(フォーマット崩れの一括修正など)を機能変更と混ぜない。
- コミットメッセージは `type: 概要` の形式を推奨する(例: `docs: CLAUDE.mdを追加`, `fix: バージイン解除タイミングを修正`)。
  `type` は `feat` / `fix` / `docs` / `refactor` / `test` / `chore` を基本とする。
- 概要は日本語・英語どちらでもよいが、**このリポジトリの既存コメント・エラーメッセージ・READMEが日本語主体**
  であることに合わせ、迷ったら日本語を選ぶ。
- `.env` や実際のトークン・APIキーなど秘密情報を含むファイルは絶対にコミットしない。
  `git add` 前に `git status` / diff内容を確認し、想定外のファイルが含まれていないか確認する。
- 動かない状態(ビルド・型チェックが通らない状態)のコミットを積み上げない。区切りの良い単位で
  `npm run typecheck` / `npm run lint` / `npm run build` を通す。

## 6. Pull Request運用

- 作業完了後、次の順で行う: テスト・検証 → コミット → Push → Pull Request作成。**マージはしない。**
- **PR本文の冒頭には必ず `## FROM：Claude Code` と明記する(全角コロン、指定文字列をそのまま使用)。**
  ユーザー・Codexなど他者の発言と混同されないようにするための表記であり、省略しない
  ([Issue #3](https://github.com/imifu/discord_viatual_friend_api/issues/3) 参照)。
- PR本文には最低限以下を含める。
  - **変更概要**: 何を・なぜ変更したか
  - **設計上の判断**: 既存パターンから外れた選択をした場合はその理由
  - **テスト計画**: 実行したコマンド(`typecheck`/`lint`/`build`等)、および該当する場合はREADMEの
    手動テスト手順のうちどの項目を実施済み/未実施か。実機Discord・実オーディオデバイスが必要で
    実施できない項目は「未実施」と明記し、実施したかのように書かない。
  - **Codexへのレビュー依頼事項**: 特に見てほしい観点があれば明記する(例: 音声ゲートのタイミング、
    秘密情報の扱い等)。
- PRは常にドラフトではなくレビュー可能な状態で作成する。中途半端な実装のままPRを出さない。
- Codexからのレビューコメントには、[7章](#7-codexとの役割分担)の方針に従って対応する。

## 7. テスト方針

現状 `package.json` に自動テストスクリプトは存在せず、README「手動テスト手順」の表が
実質的な受け入れ基準になっている。この現実を踏まえ、以下を方針とする。

- **PRを出す前に必ず実行するもの**: `npm run typecheck`、`npm run lint`、`npm run build`。
  すべてエラーなく通ることを確認してからPRを作成する。
- **実機依存の変更(Discord接続・音声デバイス・VC操作)**: README「手動テスト手順」表の該当項目を
  実際に手動実施し、PR本文に結果を書く。実行できない環境の場合は「未実施」と正直に書き、
  ユーザーに実機での確認を依頼する。動作確認したと偽らない。
- **純粋ロジックの変更(mixer/VAD/gate/ring-buffer/env検証/エラーメッセージ組み立て等)**:
  ハードウェアに依存しないため、自動テストを書くことを積極的に検討する。現在テストランナーが
  存在しないため、新規依存を避けたい方針(2章)に沿って、まずは Node.js標準の `node:test` +
  `node --import tsx` あたりの軽量な構成を提案し、ユーザーの合意を得た上で導入する。
  大規模なフレームワーク(Jest等)の導入は費用対効果をユーザーと相談してから判断する。
- テストが書けない・書く時間がない変更については、その理由をPR本文に明記し、レビュー側(Codex)が
  判断材料を持てるようにする。

## 8. Codexとの役割分担

- **Claude(私)**: 設計・実装・自動テスト整備・リファクタリング・Git操作・PR作成、および
  Codexレビューへの対応を担当する。
- **Codex**: PRレビュー・品質保証・設計レビュー・セキュリティレビュー・テスト不足の指摘を担当する
  (詳細は [AGENTS.md](./AGENTS.md)。現時点では未策定、[Issue #2](https://github.com/imifu/discord_viatual_friend_api/issues/2) 参照)。
- **Codexのレビューコメントを無条件に採用しない。** 指摘を受けたら、まず技術的に正しいか自分で検証する。
  - 指摘が妥当な場合: 修正し、対応内容をPR上で報告する。
  - 指摘が妥当でない、または意図的な設計判断による場合: 根拠を添えて反論・説明する。単に無視しない。
  - 指摘の意図が不明な場合: 実装を推測で変えず、確認を取る。
- 実装はClaude、レビューはCodexという役割分担を守り、Codexに実装を依頼したり、
  自分でレビューを完結させて役割を混同したりしない。

## 9. 禁止事項

- `main` への直接コミット
- Pull Requestの自己マージ
- 秘密情報(Discordトークン、APIキー、`.env`の中身等)をコード・ログ・コミット・PR本文に含めること
- ユーザーに確認せず要件を変更・拡大すること
- OpenAI API / Realtime APIなど、README記載の設計方針(音声デバイス経由のみでの連携)に反する実装
- Discord→GPT用とGPT→Discord用の仮想デバイスペアを共用・統合する変更
- 手動テストを実施していないのに実施したと報告すること
- 破壊的なGit操作(force push、履歴の書き換え等)をユーザーの明示的な合意なく行うこと
